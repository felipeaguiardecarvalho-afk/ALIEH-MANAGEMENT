1. RESUMO EXECUTIVO
--------------------------------------------------------------------------------
A aplicacao de gestao comercial atual e um monolito Python executado pelo
framework Streamlit. O ficheiro principal `app.py` (milhares de linhas)
concentra: autenticacao, navegacao lateral, e toda a interface dos modulos
(Painel, Produtos, Custos, Precificacao, Estoque, Clientes, Vendas, Checklist
UAT). A persistencia e PostgreSQL (tipicamente Supabase em producao), com uma
camada de servicos em `services/*` e repositorios / queries em `database/*`.

Caracteristicas distintivas do modelo Streamlit relevantes para migracao:
- Cada interaccao do utilizador pode reexecutar o script inteiro (custo de CPU
  proporcional ao tamanho de `app.py` e imports).
- Navegacao e simulada por `st.sidebar.radio` (nao sao URLs profundas por modulo
  no sentido de SPA; embora o Streamlit suporte multipagina, este projecto
  usa padrao monopagina com ramificacao por `page == …`).
- Autenticacao, tenant e RBAC vivem em `utils/app_auth.py` e `utils/rbac.py`,
  com throttling de login e auditoria.
- Cache de dados agregados no Painel via `st.cache_data` (TTL 120s / 300s em
  varias funcoes em `utils/painel_dashboard.py`).

2. ARQUITECTURA DE ALTO NIVEL
--------------------------------------------------------------------------------
2.1 Entrada e runtime
- Ficheiro: `app.py`
- Framework: `streamlit`
- Dados tabulares: `pandas`
- Graficos BI: `plotly` (no Painel executivo)
- SQL: `psycopg` (via `database/connection.py`, `prepare_threshold=0` para
  compatibilidade com PgBouncer / Supabase pooler)

2.2 Camada de negocio (nao deve ser alterada na migracao de UI pura)

  recebimento de stock, baixa manual, reset de lote, etc.

  de SKU do catalogo, razoes de bloqueio.
- `services/customer_service.py` — CRUD de clientes.
- `services/sales_service.py` — `record_sale` (transaccao atomica com stock).
- `services/read_queries.py` — camada de leitura com `st.cache_data` de curto
  TTL (10s) sobre funcoes de `database.repositories.query_repository`.
- `services/uat_checklist_service.py` — reexporta casos e funcoes UAT.
- `services/tenant_scope.py` — `effective_tenant_id_for_request` (multi-tenant).

2.3 Persistencia e schema
- `database/connection.py`, `database/config.py` — DSN, ambiente, flags.
- `database/repositories/query_repository.py` — implementacao real das queries
  de leitura (produtos, vendas, KPIs, etc.).
- `schema.sql` na raiz — referencia de schema PostgreSQL alinhado ao motor.

4. AUTENTICACAO, RBAC E MULTI-TENANT
--------------------------------------------------------------------------------
4.1 Autenticacao (`utils/app_auth.py`)
- Modos:
  (a) **Modo BD prioritario:** se existir pelo menos um registo na tabela
      `users`, apenas esse modo valida passwords (hash em `password_hash`).
  (b) **Modo legacy:** se `users` estiver vazia, credenciais em env/Secrets
      (`ALIEH_AUTH_USERNAME`, `ALIEH_AUTH_PASSWORD`, etc.).
  (c) **Desenvolvimento:** sem nenhum modo configurado, a app abre sem login.
- Producao / Streamlit Cloud: `is_production_environment()` pode forcar que
  exista autenticacao (legacy ou utilizadores na BD).
- Sessao Streamlit: chaves `alieh_auth_ok`, `alieh_auth_username`,
  `alieh_auth_user_id`, `alieh_auth_role`, `alieh_tenant_id`, contadores de
  falhas e lockout por utilizador.
- Throttling: `database/login_throttle_repo.py` — lockout apos falhas,
  `LOCKOUT_SECONDS`, `MAX_FAILURES_BEFORE_LOCKOUT`, auditoria em
  `login_attempt_audit` quando aplicavel.

4.2 RBAC (`utils/rbac.py`)
- `ROLE_ADMIN` / `ROLE_OPERATOR`.
- `require_admin()` — usado em operacoes de catalogo, custos gravados,
  precificacao, exclusoes irrevogaveis, entrada de stock (conforme docstring).
- `require_operator_or_admin()` — vendas e cadastro operacional de clientes.
- `is_admin()` / `is_operator_or_admin()` — helpers para condicionar UI;
  quando auth nao esta configurada, `is_admin()` devolve True (fluxo aberto).

4.3 Multi-tenant
- `database/tenancy.py` — `DEFAULT_TENANT_ID`, `resolve_tenant_id`.
- `services/tenant_scope.effective_tenant_id_for_request()` — todas as queries
  de leitura/escrita relevantes filtram por `tenant_id`.

IMPLICATION PARA MIGRACAO WEB:
- O prototipo Next.js deve replicar: login, sessao, tenant na cookie ou JWT,
  e as mesmas chamadas `require_admin` / `require_operator_or_admin` em cada
  rota/accao, ou delegar a RLS no Supabase com policies equivalentes.

7. MODULO — PRODUTOS
--------------------------------------------------------------------------------
7.1 Busca por SKU e lote (`st.expander` "Busca por SKU e lote")
- `services.read_queries.fetch_product_search_attribute_options()` — listas
  distintas para filtros.
- `services.read_queries.search_products_filtered(text_q, frame, lens, gender,
  palette, style, sort_by, limit, offset, tenant_id)` — devolve rows + total.
- Widgets:
  - `sku_search_text_q` — texto
  - `sku_search_sort` — sku | name | stock_desc | stock_asc
  - `sku_search_ps` — 25/50/100/200 linhas por pagina
  - Filtros: cor armação, cor lente, genero, paleta, estilo (`FILTER_ANY` +
    valores distintos)
  - `sku_search_page` — numero de pagina
  - `sku_search_product_focus` — selector de produto para painel de detalhe

7.2 Detalhe do produto seleccionado
- `fetch_product_by_id`, imagem via `product_image_abs_path`, bloqueios de
  edicao (`product_lot_edit_block_reason`, `sku_correction_block_reason`).

7.3 Cadastro de produto
- `require_admin()` em operacoes de escrita.
- Campos: nome, data de registro, atributos (dropdowns com "outro" via
  `dropdown_with_other` / `attribute_selectbox`), foto opcional
  (`st.file_uploader`).
- Accao: `services.product_service.add_product(...)` (+ bytes de imagem se
  houver).

7.4 Edicao de lote
- `st.form` por id do produto: nome, data, atributos.
- Accao: `services.product_lot_facade.update_product_lot_attributes(...)`.
- Foto: `update_product_lot_photo`.

7.5 Exclusao de SKU do catalogo
- Fluxo com confirmacao; `hard_delete_sku_catalog` quando nao bloqueado.

CONSTANTES DE LISTAS (cadastro) — ver `app.py`:
- `PRODUCT_GENDER_OPTIONS`, `PRODUCT_PALETTE_OPTIONS`, `PRODUCT_STYLE_OPTIONS`,
  `PRODUCT_FRAME_COLOR_OPTIONS`, `PRODUCT_LENS_COLOR_OPTIONS` (listas longas).

16. LIMITACOES INERENTES AO MODELO STREAMLIT (RELEVANTES PARA MIGRACAO)
--------------------------------------------------------------------------------
- Reexecucao completa do script por interaccao (latencia e custo de servidor).
- Navegacao sem URLs profundas por modulo (partilha de links para ecra exacto
  limitada vs App Router).
- Upload de ficheiros (`st.file_uploader`) — em Vercel/serverless exige storage
  externo (Blob, Supabase Storage, S3).
- Debug de BD visivel no arranque — risco operacional se exposto publicamente.
