1. RESUMO EXECUTIVO
--------------------------------------------------------------------------------
A camada web de protótipo está dividida em dois pacotes isolados no mesmo
repositório, sem alteração ao Streamlit de produção nem aos serviços Python
legados como ponto de entrada:

  • /web-prototype   — Next.js (App Router), TypeScript, Tailwind, UI estilo
                       shadcn, Server Components / Server Actions.
  • /api-prototype   — FastAPI que importa `services/*` e `database/*` existentes
                       (mesma lógica de negócio que a app Streamlit).

Regra de isolamento: código novo de protótipo restringe-se a essas pastas;
objetivo é replicar comportamento, não substituir produção ainda.

Fluxo predominante de escrita: Next.js (Server Actions) → HTTP com cabeçalhos
de contexto (inquilino, utilizador, perfil) → api-prototype → serviços /
repositórios Python já existentes.

Leituras: parte das páginas já consome apenas a API de protótipo; outras ainda
usam `lib/queries.ts` (SQL server-side no Next via `postgres`) para listagens
legadas ou rotas auxiliares — ver secção 5 (matriz híbrida).

Autenticação no protótipo: login com JWT em cookie httpOnly (`AUTH_SESSION_SECRET`),
tabela `users` em PostgreSQL (ou credencial legacy em desenvolvimento), perfis
`admin` / `operator` / `viewer`, tenant via `ALIEH_TENANT_ID` e JWT. Middleware
protege rotas (exc. modo aberto `ALIEH_PROTOTYPE_OPEN=1`).

3. API DE PROTÓTIPO (`/api-prototype`)
--------------------------------------------------------------------------------
Prefixos REST (todos com contexto `X-Tenant-Id`, `X-User-Id`, `X-Role`; opcional
`X-Username` para trilho de auditoria / UAT):

  GET/POST …        /health
  …                 /sales/record, /sales/product-context/{id}, /sales/recent
  …                 /dashboard/filters, /dashboard/panel
  …                 /products, /products/{id}, atributos, SKU, custos, imagem
                    pública (URL Supabase validada), etc.
  …                 /customers, /customers/{id}
  …                 /inventory/lots, write-down, stock-receipt, batches/exclude
  …                 /pricing/* (sku-master, snapshot, workflow, histórico, …)
  …                 /uat/records (leitura; permite viewer), /uat/upsert (mutação)
  …                 /storage/signed-upload (mint de URL; sem bytes de ficheiro
                    na API)
  …                 /audit/events, /audit/login-ingest (trilho de atividade)

Autenticação na API: perfis `admin` e `operator` na maioria dos endpoints;
`get_actor_read` permite `viewer` apenas onde necessário (ex.: GET /uat/records).

Persistência de auditoria de protótipo: tabela `prototype_audit_events` criada
com `CREATE TABLE IF NOT EXISTS` no arranque da API (eventos de vendas,
precificação, stock, login/logout conforme implementação atual — ver código).

5. SERVER ACTIONS (escritas) — RESUMO
--------------------------------------------------------------------------------
Todas as mutações abaixo delegam em `apiPrototypeFetch` para a api-prototype
(exceto onde indicado):

  customers.ts     POST/PATCH/DELETE /customers
  products.ts      POST /products, PUT atributos, DELETE SKU, reset de lote
  costs.ts         POST custos por SKU
  pricing.ts       POST workflow de precificação (+ leituras auxiliares na API)
  inventory.ts     write-down, stock-receipt, exclusão de lotes
  sales.ts         contexto de produto + registo de venda
  uat.ts           POST /uat/upsert
  auth.ts          sessão JWT; ingestão opcional de auditoria de login
  product-image-upload.ts  POST /storage/signed-upload (metadados apenas)

Gates: `gateMutation()` / `requireAdmin()` / `requireOperator()` conforme rota.

6. FUNCIONALIDADES RECENTES (DESDE RELATÓRIOS ANTERIORES)
--------------------------------------------------------------------------------
• Painel: filtros avançados, dados agregados via API, mantendo `revalidate = 120`
  e `Suspense`.
• Upload de imagem de produto: Supabase Storage, fluxo URL assinada → cliente →
  gravação de URL pública validada na API (`product_image_storage_url`).
• Auditoria de protótipo: eventos persistidos em `prototype_audit_events` +
  trilho de login pré-sessão com segredo partilhado (`PROTOTYPE_AUDIT_INGEST_SECRET`
  documentado em `.env.example`).
• UAT: leitura e escrita só pela API; utilizador de apresentação = nome de
  sessão (`X-Username`) ou fallback para id; perfil = `X-Role`; viewers podem
  consultar o checklist.

9. RISCOS E RECOMENDAÇÕES
--------------------------------------------------------------------------------
1. Dupla origem de leitura (SQL em `queries.ts` vs API) — risco de divergência de
   filtros, tenant e cache. Recomendação: migrar as páginas restantes (`/sales`,
   `/costs`, `/api/batches`) para endpoints da api-prototype alinhados às queries
   Python existentes.

2. DDL em arranque (`prototype_audit_events`) — aceitável em ambiente de
   protótipo; validar política de migrações em produção antes de qualquer
   promoção.

3. Segredos: `PROTOTYPE_AUDIT_INGEST_SECRET` e `SUPABASE_SERVICE_ROLE_KEY` devem
   permanecer apenas no servidor; nunca no bundle cliente.

4. Testes E2E (Playwright) contra staging com os casos UAT-M-* continuam
   recomendados antes de substituir a UI Streamlit.
