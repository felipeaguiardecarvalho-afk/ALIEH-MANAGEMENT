RESUMO EXECUTIVO
--------------------------------------------------------------------------------
  • A aplicação usa **exclusivamente PostgreSQL** via ``database.connection.get_db_conn``.
    Não há fallback silencioso para SQLite em falha de ligação: excepção
    ``ConnectionError`` com mensagem ``PostgreSQL connection required but failed`` e log
    FATAL sem credenciais.

  • ``get_postgres_conn``: DSN resolvido com ``get_database_url()`` (env + segredos Streamlit)
    depois ``get_supabase_db_url`` / ``get_postgres_dsn``. ``psycopg.connect`` com
    ``prepare_threshold=0`` (kwarg e ``conn.prepare_threshold = 0`` após abrir), ``sslmode``
    no DSN (``require`` quando ausente; preserva ``verify-full`` se já no URL),
    ``autocommit=True``, ``row_factory=dict_row``, ``connect_timeout`` (defeito **10** s;
    override ``DATABASE_CONNECT_TIMEOUT``), ``hostaddr`` IPv4 quando aplicável. **Sem** comandos
    SQL de sessão após ligar (não ``DISCARD ALL`` / ``SET SESSION``) — mitiga
    ``DuplicatePreparedStatement`` no pooler. Ajuste **client-side**: ``cursor(..., binary=False)``.
    Na primeira ligação bem-sucedida (não ``silent_probe``): INFO
    ``PostgreSQL connection established (PgBouncer safe mode)``.

  • **SQLite** permanece para: ``init_db`` quando ``DB_PROVIDER=sqlite`` (DDL incremental),
    export/migrações e testes — ligação via ``database.sqlite_tools.sqlite_connect_for_local_schema``
    e path ``DB_PATH`` (``/data``, ``.alieh_data`` ou temp). Não é o motor da app em produção
    com ``DATABASE_URL``.

  • Logs de banco: backend activo, tempos de query Postgres em ``sql_compat`` (INFO/WARNING
    por limiar), erros com prévia de SQL; limiares ``ALIEH_DB_QUERY_INFO_MS`` / ``ALIEH_DB_QUERY_WARN_MS``.

  • Modo shadow: replica escritos Postgres para SQLite espelho (opcional).

  • Ferramentas: schema_apply, schema_validate, cross_db_parity, health_check;
    integração Postgres opt-in (``ALIEH_PG_INTEGRATION``).

3. MOTOR DE BASE DE DADOS
--------------------------------------------------------------------------------
  Ficheiros: database/config.py, database/connection.py, database/sql_compat.py,
  database/sqlite_tools.py, database/init_db.py

  • **get_db_conn** → ``return get_postgres_conn()``; qualquer excepção → log
    ``FATAL: PostgreSQL connection failed — no fallback allowed`` e
    ``ConnectionError("PostgreSQL connection required but failed")``. Sem SQLite.

  • **Queries Postgres:** ``db_execute`` / ``run_insert_returning_id`` usam
    ``conn.cursor()`` + ``execute(..., prepare=False)`` — evitar ``Connection.execute`` no
    pooler.

  • **health_check / schema_apply / production_readiness_check** e outros módulos: probes e
    DDL com ``prepare=False`` onde aplicável (ver ficheiros).

  • **fetch_product_search_attribute_options:** DISTINCT + ORDER BY compatível Postgres
    (subquery externa).

  • **Provedor (config):** ``get_database_provider`` ainda distingue sqlite vs postgres para
    ``init_db`` e ferramentas; não altera o facto de ``get_db_conn`` ser só Postgres.

  • **init_db:** só corre DDL SQLite se ``get_db_provider() == "sqlite"``; lê ``DB_PATH`` em
    runtime via ``connection_module.DB_PATH``. Em Postgres, assume schema já aplicado
    (schema.sql / Supabase).

  • **Streamlit Cloud / backups SQLite:** ``utils/audit_backup`` pode usar caminho gravável
    alternativo a ``/backups`` quando sem permissão.


4. OBSERVABILIDADE — QUERIES E ERROS
--------------------------------------------------------------------------------
  • SQLite (TimedSqliteConnection): duração e ERROR em falhas (caminhos dev/export).

  • Postgres (sql_compat): registo de duração e falhas com ``exc_info``.

  • Limiares (ms): ALIEH_DB_QUERY_INFO_MS (100), ALIEH_DB_QUERY_WARN_MS (1000).


5. MODO SHADOW
--------------------------------------------------------------------------------
  database/shadow_mode.py — espelho SQLite em runtime **só** com os três: ALIEH_DB_SHADOW_MODE=1,
  ALIEH_DB_SHADOW_SQLITE=caminho, e ALIEH_ALLOW_RUNTIME_SQLITE=1 (sem este último, não há
  ``sqlite3.connect``; um WARNING por processo). Ver docstring para limitações.


6. SCHEMA E PARIDADE
--------------------------------------------------------------------------------
  database/schema_apply.py — aplica ``schema.sql`` ao Postgres; CLI:
    • ``python -m database.schema_apply`` — só DDL do ficheiro;
    • ``python -m database.schema_apply --full-reset`` — ``DROP SCHEMA public CASCADE`` + DDL completo
      (detecta Supabase pelo DSN; grants ``anon``/``authenticated``/``service_role`` após tabelas);
    • ``--supabase`` / ``--no-supabase-grants`` — forçar ou omitir grants API.
    Preferir URI **directa** Supabase (porta 5432) para DDL pesado; **apaga todos os dados** em ``public``.
  validate_postgres_schema — database/schema_validate.py
  compare_sqlite_postgres — database/cross_db_parity.py (ligação dedicada; cursor + prepare=False)
  migrate_sqlite_to_postgres.py — caminhos absolutos; opcional --full-reset (migração de dados)
